# PRD：YT Rec v2（依實際使用情境重新定位）

> 版本：v2 草案　日期：2026-06-15　撰寫：Claude (Opus 4.8)
> 前身：`PRD-LiveClipFast.md`（v1）。本版依 owner 釐清的真實使用情境重寫定位。
> v1 已實作並真機驗證的部分（雙軌引擎、SCK 擷取、快剪、災難復原、Developer ID 簽名）大量沿用，**不是打掉重練**。
> 標 ⚠️ 的是「我做的設計決策／假設，owner 可推翻」。

---

## 1. 改版緣由（owner 的真實情境）

v1 把「**下載原生檔**」當主線、螢幕側錄當保險。但 owner 的實際用法是反過來的：

1. **直播太長，載下來太麻煩**（6–8h、24hr 台）→ 下載不切實際。
2. **正在直播中**就要錄 → 等不到 VOD。

→ 所以實際主力是「**螢幕錄影式擷取**」：錄下直播的影像＋聲音。配套的硬需求：
- **要能繼續用電腦工作**（傳統螢幕錄影佔滿螢幕、不能動電腦——這是要解的痛）。
- **不能錄到其他 App 的聲音**（Premiere 音效、LINE 通知…）。
- **要看得到畫面，確認在錄對的內容**（v1 用隱藏視窗 → 使用者完全看不到在錄什麼，是最大缺口）。

**一句話新定位**：把一個 YouTube 直播視窗「單獨」錄下來（影像＋該視窗的聲音），**看得到、聽得到、但不佔用你的工作、也不會把你電腦其他聲音錄進去**。

## 3. 核心價值（vs 傳統螢幕錄影）

| | 傳統螢幕錄影（QuickTime 等） | YT Rec v2 |
|---|---|---|
| 錄什麼 | 整個螢幕 | **只錄那個直播視窗**（SCK 單一視窗擷取） |
| 能不能同時工作 | 不能（整螢幕被佔／被錄） | **能**：其他視窗疊在上面也不會被錄進去 |
| 其他 App 聲音 | 會一起錄進去 | **不會**（SCK 只收該視窗 App 的音訊，已實測） |
| 看得到在錄什麼 | 看得到（但就佔螢幕） | **看得到**（可見、可縮放、可置頂的小監看視窗） |
| 超長直播 | 檔案爆大、要顧 | fMP4 抗崩潰切片＋時長／磁碟保護，可長錄 |

## 4. 關鍵設計決策（⚠️ = 我的決策，可推翻）

**D1 ✅【owner 已確認 2026-06-15】擷取視窗改「可見、緊湊、浮動、可置頂、可縮放」（即時，不延遲）**
v1 是隱藏離屏視窗（看不到）。v2 改成一個**可見的小視窗**播放直播，使用者直接看到內容（即時，不延遲）。owner 在「即時可見 / 延遲預覽 / 兩者可切換」三選一中選定**即時可見小視窗**。
- 預設緊湊（如 640×360 或 854×480），可拖到角落／外接螢幕，可選「總在最上層」。
- **關鍵**：SCK 用 `desktopIndependentWindow` 只擷取「這個視窗的內容」——使用者把 Premiere 等疊在它上面，**疊上去的東西不會被錄進去**（v1 §6.3-3 已驗證離屏/遮蔽照錄）。所以「看得到」與「不錄到工作畫面」可同時成立。
- 可一鍵「縮到背景（僅錄聲音與畫面、不顯示）」——不想盯著時收起來，擷取續跑（即 v1 的隱藏模式，當成可切換的狀態）。
- ⚠️ 擷取**畫質與視窗顯示大小解耦**：畫質在設定選（720p/1080p），縮小監看視窗不影響錄製畫質（實作上：視窗以目標解析度算繪，顯示可縮放；需驗證 SCK 對小視窗的擷取行為）。

**D2 ⚠️ 螢幕側錄為主、下載為輔（依 live_status 智慧預設）**
- `is_live`／`is_upcoming`（進行中）→ **預設只螢幕側錄**（下載長直播不切實際）。
- `post_live`／`was_live`／`not_live`（已是有限長度 VOD）→ **建議／預設改用下載**（更快、畫質更好、不用即時錄）；側錄當退路。
- 下載軌保留但降為「選項」，不再是預設主線。馬拉松判定併入此邏輯（進行中且超長 → 不下載）。

**D3 ⚠️ 砍掉 Core Audio process tap，音訊只走 SCK**
- owner 接受有聲 → 不再需要 tap 的「靜音」。而 tap 那條：(a) 收不到 WebKit GPU 子行程音訊、(b) 修法依賴的私有 API 在簽名版會 SIGSEGV（本輪實證）。**整條移除**（ProcessAudioTap、音訊來源決策窗、§7.5 race、自我檢測測試音）。
- 音訊改純 SCK（`capturesAudio` on 單一視窗 filter）——**已實測：只收該視窗 App 音訊、不含其他 App（afplay 1kHz 注入零殘留）**、可解碼、雙軌齊。
- 連帶簡化：移除「靜音/有聲」模式切換與相關 UI、log、通知文案。
- **真靜音（喇叭也不響）= v-next**：內嵌虛擬音訊裝置（類 BlackHole）把該視窗輸出導到「靜音但可擷取」的裝置。

**D4 權限三件套 → 兩件套（⚠️ 待驗證）**
- 砍 tap 後不再需要「系統音訊錄製」TCC（那是 process tap 要的）。SCK 的音訊擷取屬「螢幕錄製」權限範疇。
- → 預期只需 **螢幕錄製 ＋ 通知** 兩個權限。⚠️ **須真機確認**：移除 tap 後，SCK `capturesAudio` 不需要額外的系統音訊權限。

**D5 即時監看與打點剪輯分工**
- **監看（看內容）**＝ D1 的可見擷取視窗（即時、有聲、確認在錄對的東西）。
- **打點剪輯（剪精華）**＝ 沿用 v1 的本機 HLS 播放器（延遲 6–8s 反而利於打點），對「已錄到的切片」標 I/O → ⌘E 快剪。監看視窗靜音與否不影響（打點用的是已落盤切片）。

## 5. 功能規格

### 5.1 主面板（控制中心）
- 網址輸入＋貼上鈕＋「立即開錄」主鈕。
- 進行中任務卡：標題、擷取狀態（`錄製中・00:12:34`、畫質）、按鈕「顯示／隱藏監看視窗」「打點剪輯」「停止」。
- live_status 提示：偵測為「進行中直播 → 螢幕側錄」或「已結束影片 → 建議下載」（D2）。
- 最近成品（最多 5）：整列可拖進 Premiere；剪刀（打點）、放大鏡（Finder）。
- 權限缺失頂部警告列＋「前往授權」。
- 底部：打開輸出資料夾、Log。

### 5.2 即時監看視窗（NEW，取代隱藏視窗）
- 可見、緊湊、可拖移、可縮放、可選「總在最上層」。播放目標直播（即時、有聲）。
- SCK 擷取此視窗 → 其他視窗疊上去不影響錄製內容。
- 「縮到背景」鈕：隱藏畫面、擷取續跑（給不想盯著的長錄場景）。
- 角落顯示紅色錄製指示＋時長。
- ⚠️ 注入 JS 沿用 v1（防背景暫停、強制播放、`ended` 事件回報）；但因視窗現在可見/前景，背景節流風險大降（v1 §7.7 的離屏問題從源頭緩解）。

### 5.3 預覽確認（純播放，不在 App 內剪接）
> 定位：**「錄什麼有什麼」**——成品就是交付物，剪接在 Premiere 做。App 內只負責「播來確認錄對了」，不做更弱的內建剪輯（2026-06-18 owner 拍板，移除打點剪輯與 `ClipExporter`）。
- 對已完成檔案用 `AVPlayerView`（內建拖曳控制軸）播放，確認**錄對了串流、聲音在、畫面沒問題**。
- 不提供 I/O 打點、⌘E 快剪、`精華片段/` 抓段——要剪接直接把成品從主面板**拖進 Premiere**。
- 「在 Finder 顯示」可定位成品檔。
- ⚠️ `AVPlayerView` 預覽在 release 包的播放穩定性仍建議真機點測一次。

### 5.4 音訊（純 SCK，已驗證乾淨）
- `SCStreamConfiguration.capturesAudio = true`，filter＝單一監看視窗 → 只收該視窗 App 音訊。
- 48kHz 立體聲 AAC，與影像同 host-time 時鐘對齊。
- **不收其他 App 聲音**（已用 afplay 1kHz 注入測試證實零殘留）。

### 5.5 設定頁（精簡）
1. 輸出資料夾（預設 `~/Movies/YT-Rec`）
2. 擷取畫質：1080p30・8Mbps／720p30・5Mbps
3. 監看視窗：總在最上層（開/關）、開錄時自動顯示（開/關）
4. 時長上限：3h／6h（預設）／12h／不限
5. 下載軌：關（預設，僅進行中直播）／自動（依 live_status）／永遠嘗試
6. 已結束影片到手後的側錄檔：丟垃圾桶（預設）／保留
（＋打開 Log 資料夾）

### 5.6 可靠性（沿用 v1，已驗證）
- App 結束前保檔（terminateLater）。
- **災難復原**（已真機驗證生效）：啟動掃 `.work/segments/`，未收工自動拼回 MP4＋通知。
- 防睡眠（beginActivity）。
- fMP4 切片天生抗崩潰。
- 系統通知：開錄、收工保存、磁碟/時長保護、精華匯出、災難復原。

### 5.7 任務資料夾（沿用）
```
~/Movies/YT-Rec/20260615-2230 <ID或標題>/
  側錄_<標題>.mp4            ← 螢幕側錄成品（主）
  <yt-dlp 原生檔>.mp4         ← 下載成品（若啟用）
  精華片段/精華_001530-001745.mp4
  .work/segments/            ← fMP4 切片（任務關閉清除）
```

### 5.8 磁碟保護（沿用 v1，已單元測試）
< 15GB 警告、< 8GB 拒錄；側錄中每 60s 檢查 < 10GB 自動收工保檔。

## 9. 自我 QA（設計審視，不是橡皮圖章）

- **Q：可見視窗播音訊 → 長錄數小時，使用者耳朵一直聽到直播聲，累不累？**
  A：真實成本，誠實揭露。緩解：可「縮到背景」（畫面收起、聲音仍在，因 SCK 要它出聲才錄得到）。**真靜音（喇叭也不響）只能等 v-next 虛擬裝置**。若 owner 覺得長錄必須靜音，v-next 要提前。
- **Q：縮小監看視窗會不會降畫質？**
  A：⚠️ 風險點。需驗證 SCK 對「顯示縮小但內容算繪為 1080p」的視窗擷取行為；最壞情況＝畫質跟著視窗大小，那就改「畫質＝視窗內容尺寸」並提示使用者「視窗越大畫質越好」。**實作前先做 SCK 小實驗確認**。
- **Q：SCK 單一視窗音訊，真的只收該 App？多來源（同時 Music＋LINE＋Premiere）都不漏？**
  A：本輪只用單一 afplay 1kHz 驗證乾淨。**T2 要多源再驗**（同時三個聲音源）才算數。
- **Q：砍 tap 後，SCK 音訊是否仍需「系統音訊錄製」權限？**
  A：⚠️ 架構上應只需螢幕錄製，但本輪是兩個權限都開著測的，沒隔離驗證。**D4 必須真機隔離確認**，否則 Info.plist 與權限引導會錯。
- **Q：可見視窗 → YouTube 還會不會背景節流音訊解碼（v1 §7.7 的鬼）？**
  A：可見/前景視窗理論上不被背景節流，風險比離屏低；但仍須真機確認 `webkitAudioDecodedByteCount > 0`（前景版）。
- **Q：下載降為輔，會不會讓「已結束短片」體驗變差（變成用螢幕錄）？**
  A：D2 用 live_status 智慧切換解決——VOD 預設走下載。需確保探測準確。
- **Q：總在最上層的小視窗，在全螢幕 Premiere 會不會擋到？**
  A：「總在最上層」設為**可關**；預設可考慮關，由使用者自己擺到外接螢幕/角落。
